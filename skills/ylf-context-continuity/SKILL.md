# 上下文连续性闭环

这是一个用户主动调用的三段式流程：`handoff` 保存任务状态，`reflect` 分析协作纠正，`remember` 把可复用教训写入经验记录或系统记忆。用户指定模式时只运行对应模式；用户要求完整闭环时按这个顺序运行。

## 共同规则

1. 先读取当前会话、相关文件和现有记录，再写入任何内容。当前会话中的用户约束、已确认事实和最新代码优先。
2. 只记录有证据的内容。无法确认的事项写成待确认项，并标明需要检查的文件、命令或用户决定。
3. 每个模式都要给出可检查的完成结果：文件路径、写入的章节或表格、未解决项，以及仍需用户决定的事项。
4. 保持`单一事实来源`：更新已有的同类记录，合并重复内容，把同一经验只保留在最合适的位置。

## `handoff`：写任务交接

适用于长会话结束、跨天推进或项目中间状态复杂的任务。

### 执行

1. 确定项目根目录：优先使用当前任务实际工作的目录；若存在多个候选目录，使用包含当前改动和主要项目文件的目录。交接文件固定命名为该目录下的 `HANDOFF.md`。
2. 读取已有 `HANDOFF.md`（如果存在），保留仍然有效的上下文，删除已经过时的状态，并合并重复条目。
3. 用下面的结构写清当前任务。每项都以当前会话或文件证据为依据：

```md
# 任务交接

## 我们在做什么

## 已经完成

## 当前状态与卡点

## 下一步计划

## 任务相关的坑

## 关键文件与验证结果
```

4. `任务相关的坑`只写本任务中会影响后续执行的具体事实，例如接口契约、运行时边界、失败命令和有效 workaround。通用经验放到`reflect`或`remember`，不要堆进交接文件。
5. 下一步计划按执行顺序排列，并让每一项包含明确目标和验收信号；未完成项不能写成已完成。

完成标准：`HANDOFF.md` 位于正确项目根目录，六个章节均已填充或明确标记“无”，当前卡点和下一步计划可被无上下文的新会话直接执行，且不包含未经确认的结论或泛化经验。

## `reflect`：复盘协作纠正

适用于用户指出偏差、否定根因、补充约束，或需要检查本轮判断质量的场景。

### 执行

1. 从当前会话逐条找出用户实际纠正过的输出、判断、范围或流程。没有明确纠正时不要凭空补充；输出一行“未发现本轮已确认的纠正”。
2. 对每条纠正归因：
   - `信息不足`：当前上下文缺少参考实现、约束、环境事实或验收标准，导致合理信息无法得出结论。
   - `判断逻辑问题`：必要信息已经存在，但仍然误读证据、跳过验证、扩大范围或选错了执行顺序。
   - 两者同时存在时，先写缺失信息，再指出逻辑失误。
3. 为每条纠正生成一句下一轮开场指令建议。建议必须补齐具体的参考对象、边界、验证顺序或停止条件，不写空泛的“请更仔细”。
4. 输出并保存以下三列表格；列名保持不变：

| 我修改的内容 | 归因 | 下次的开头指令建议 |
| --- | --- | --- |
| 用户纠正的具体内容 | 信息不足 / 判断逻辑问题 / 两者 | 可直接复制的开场约束 |

5. 查找项目现有的经验记录文件，优先使用已有的 `LESSONS.md`、`经验记录.md`、`错题本.md` 或项目约定文件。若用户明确要求保存且没有现成位置，在项目根目录创建 `经验记录.md`，并把本表格追加到带日期的复盘章节。

完成标准：表格每一行都对应当前会话中的真实纠正，归因能解释“缺什么”或“错在哪里”，开场建议可直接执行，表格已保存到确定的经验记录路径或明确报告保存阻碍。

## `remember`：沉淀长期经验

适用于用户明确要求“记住”“录入错题本”或“写入系统记忆”时。没有明确授权时只完成复盘，不写入长期记忆。

### 执行

1. 读取目标经验库的已有条目，按关键词、文件、症状和解决方法查找相似记录。
2. 只提炼两类内容：用户纠正形成的规则，以及已经验证有效的方法。当前任务状态、普通操作步骤和 skill、`AGENTS.md` 已明确的规则不写入。
3. 发现类似条目时更新旧条目，合并更准确的触发条件、证据和有效做法；只有确实是新规律时才新增条目。
4. 若使用本机系统记忆，遵循当前环境的记忆写入规则：向 `C:\Users\Immerse\.codex\memories\extensions\ad_hoc\notes\` 新增一个小而独立的 Markdown 更新笔记，不直接改写主记忆文件。文件名使用时间戳和简短主题，例如 `20260722-ylf-context-continuity.md`。
5. 写入后重新读取目标记录，确认没有重复条目、没有把一次性状态误写成长期规则，并报告实际写入位置和更新内容摘要。

完成标准：每条沉淀都能追溯到本轮纠正或有效方法；相似经验已合并；目标记录可重新读取；长期记忆写入遵循平台规则，或清楚报告无法写入的原因。

## 完整闭环

用户要求“按这套流程处理”或同时要求交接、反思和记忆时，依次完成：

1. 先完成 `handoff`，只保留任务进度。
2. 再完成 `reflect`，产出三列表格并保存经验记录。
3. 最后在用户明确授权时执行 `remember`，更新错题本或系统记忆。

每一步完成后再进入下一步；最终报告分别列出 `HANDOFF.md`、复盘表格和长期记忆的实际路径及未解决项。

## 边界

- `HANDOFF.md` 是交接状态文件；经验记录和系统记忆是长期资产，三者用途分开。
- 交接内容面向没有上下文的新会话，使用事实、路径、命令和验收信号；反思内容面向改进下一轮协作，使用纠正、归因和开场指令。
- 反思必须区分信息缺失与判断失误；不能用“信息不足”掩盖已经具备证据却没有验证的判断。
- 对删除、批量改写、提交代码或其他超出记录范围的动作，先遵守当前项目和用户的明确约束；本 skill 不自行扩展这些动作。
